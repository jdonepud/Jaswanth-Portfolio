CS-001 Security Automation & AWS Security Hub Reporting @ Nokia python · aws security hub · ci/cd ▸

The Problem

Security audit logs and scan output arrived faster than the team could validate them, and reporting was periodic and manual, so findings lived far longer than policy allowed.

My Role

Security Automation Engineer owning the automation track: design and build Python tooling for audit-log analysis and security validation, and integrate it into the team's AWS and CI/CD environment.

Architecture

audit logs ──▶ collect & parse (python) ──▶ analysis ──▶ prioritized findings AWS accounts ──▶ Security Hub ──▶ automated reporting ──▶ faster remediation telemetry (JSON) ──▶ REST API integrations ──▶ security operations git push ──▶ CI pipeline ──▶ security gates ──▶ fail build on critical findings

Technical Decisions

  • Parse and deduplicate before routing: the same finding across many hosts is one decision, not many tickets.
  • Severity thresholds gate CI builds; lower findings report asynchronously, so pipelines stay fast.
  • Security Hub as the single pane: aggregate and report at the source rather than teaching everyone a new tool.

Security Considerations

  • Automation itself ran with least privilege: read-only credentials, scoped per account.
  • Findings data treated as sensitive: access-controlled, never exported to unmanaged systems.

Outcome

manual validation −60%testing time −40%security shifted left into CI

Lessons Learned

Adoption beats elegance. The tooling that survived was the version that fit the team's existing workflow, not the one with the cleverest design.

CS-002 API & AppSec Testing for Bug-Bounty Programs appsec · oauth · python recon ▸

The Problem

Enterprise bug-bounty programs (Meta, Mercedes-Benz) reward finding what automated scanners miss: authorization gaps, business-logic flaws, and API abuse in real production systems.

My Approach

Treat each target as a threat-modeling exercise first, tooling second. Map trust boundaries, then probe the seams: OAuth flows, access control, session handling, and privilege escalation.

Technical Decisions

  • Built Python tooling to automate reconnaissance, vulnerability validation, and evidence collection, so time went to analysis, not setup.
  • Focused on business-logic and authz classes, where scanners are weakest and impact is highest.
  • Reported to OWASP and responsible-disclosure standards, with reproduction steps a developer can act on.

What It Sharpens

  • The attacker's view of the same systems I defend: it makes my hardening decisions concrete.
  • Writing findings that get fixed, not just filed.
CS-003 SpeedPulse · Network Performance Diagnostics networking · javascript · real-time analytics ▸

The Problem

"The Wi-Fi is slow" is a useless bug report. I wanted a tool that turns a user complaint into real network data: throughput, latency, and jitter, measured over time on both mobile and Wi-Fi.

My Role

Sole engineer: protocol-level measurement design, client app, and the real-time analytics view.

Technical Decisions

  • Parallel connections to saturate the link for honest throughput numbers; single-stream tests understate real capacity.
  • Repeated small-payload round trips to separate latency from bandwidth, since they fail differently and mean different problems.
  • Historical tracking, because one measurement is an anecdote; a trend is a diagnosis.

What It Taught Me

  • TCP behavior in the wild: slow start, congestion windows, and why the first second of a speed test lies.
  • Last-mile networks are messier than any lab; measurement tooling must assume noise.

CS-004 Memoir · Secure Personal Data Vault django · authn · secure storage ▸

The Problem

Personal memories (notes, photos, private thoughts) are exactly the data users trust apps with and apps most often leak. I built Memoir to practice treating user data like it matters.

Security Considerations

  • Per-user data isolation enforced at the query layer; no object is accessible by guessing an ID.
  • Django's session and CSRF protections kept on and configured, not fought against.
  • Input validated at the boundary; stored content escaped on render.

Lessons Learned

  • Framework defaults are a security feature; most real-world Django vulns come from disabling them.
  • Threat modeling a "harmless" notes app finds the same classes of bugs as any enterprise system: authz, injection, exposure.

Outcome

secure-by-default CRUD appauthz enforced per object A small app, deliberately built the way big apps should be.
